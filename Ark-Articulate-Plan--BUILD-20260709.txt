# Plan: The Ark + Articulate (WTC)

**Source of truth:** `~/Desktop/Whinfell_Transmission_Control/Ark-Articulate-Plan-20260709.txt`  
**Method:** `/plan` + `/arena` (4 explore specialists: Data, Shell, Clipboard/UX, Test/Build)  
**Scope:** Phases 1–3 only. Phase 4+ nice-to-haves stay deferred.

---

## Context

Today multiple modules independently `fetch` the same raw files (`latest.json` triple-loaded on refresh; BasisWatch also loads curve history; BBDM HTML loads its own report). `Agents.md` already declares **The Ark** as the sole raw-data loader, but `js/ark.js` does not exist yet.

**Articulate** is a composable intelligence layer: generate a floor-trader prompt from the current panel’s data, copy to clipboard, and log the session. There is no Articulate engine yet (BBDM only has a placeholder “articulator” rail slot).

**Why change:** enforce single source of truth, stop dual-fetch races, give operators an ARK status page + “Fix with LLM” diagnostics, and add the A analysis workflow without regressing BasisWatch / Bang Bang DA / desk refresh.

---

## Arena consensus (key decisions)

| Topic | Decision | Rationale |
|-------|----------|-----------|
| **Ark boundary** | Ark = **load + cache + stamp + subscribe**. Apply-to-UI stays in `core.hydrateFromBundle` / BW `mergeHydrationBundle` / WMC merge initially | Avoid rewriting the entire hydrate path in one PR; matches “SSOT for files” without boiling the ocean |
| **ARK / A placement** | Dig specialized tools (same path as BasisWatch / Midwest), **not** Scan view-shortcuts | Full-canvas takeover; product asks for full pages |
| **Collapsed labels** | Literal **`ARK`** and **`A`** via CSS `::before` | Product requirement (not `AK`) |
| **Nav section** | New left-rail section **“Intelligence”** (or append under Tools after Midwest) with `#btnIaArk` + `#btnIaA` | Keeps Tools clutter down; either is fine — prefer **Tools** first for minimal HTML churn |
| **Articulate module** | New `js/articulate.js` (`window.WTM_Articulate`) shared by main desk + BBDM | Separate from Ark; pure prompt + log + clipboard |
| **Log storage** | `localStorage` key `wtm_articulate_log_v1` (JSON array, capped, try/catch) | Matches prefs/theme patterns; Phase 4 can add per-user names later |
| **A buttons (in-scope major panels)** | BasisWatch utility cluster + BBDM scan toolbar next to Export; optional other dig panels only if time after 3.x core | Product: “major panels” = BW + BBDM first |
| **BBDM rewire** | Standalone `bang_bang_da_machine.html` must load `js/ark.js` and route report/hydration meta through Ark | Shell JS has no loaders today; loaders live in HTML |
| **Out of scope** | Phase 4+ (user name, summaries, multi-select, review templates, Guru, search/filter) | Explicit in product plan |

## Recommended approach (phased)

Implement **strictly in product order**. Do not start Phase 2 until Phase 1 is verified; same for Phase 3.

### Phase 1 – The Ark

#### 1.1 Create `js/ark.js` (central data engine)

**Public API** (`window.WTM_Ark` or `window.TheArk` — prefer **`WTM_Ark`** to match `WTM_*` globals):

```text
WTM_Ark.loadHydration({ force })  → { ok, data, as_of, snapshot_id, freshness_status, error? }
WTM_Ark.getHydration()            → cached bundle | null
WTM_Ark.getStamp()                → { as_of, snapshot_id, freshness_status, lastRefreshedAt, sources[] }
WTM_Ark.loadCurveHistory({ force }) → { ok, data, … }
WTM_Ark.getCurveHistory()
WTM_Ark.loadBbdmReport({ force }) → { ok, data, … }   // report JSON (+ optional API later)
WTM_Ark.getSources()              → inventory for ARK page
WTM_Ark.subscribe(listener)       → unsubscribe fn; fires on successful reload
WTM_Ark.buildFixWithLlmPrompt()   → diagnostic string for clipboard
WTM_Ark.BUILD
```

**Behavior:**

- Sole owner of raw `fetch` for:
  - `data/hydration/latest.json`
  - `data/barchart/v1/barchart_curve_history.json`
  - `bang_bang_da/bang_bang_da_report.json` (and path cascade used today)
- In-memory cache + `force` bust (`?_=${Date.now()}`, `cache: 'no-store'`) matching current BW/core patterns
- Source registry entries: id, url, status (`ok|missing|error|stale`), last success time, error message
- No DOM rendering inside `ark.js` (Lego: data only)

#### 1.2 Full ARK page (Dig tool)

Mirror Midwest dig embed:

| Piece | Action |
|-------|--------|
| `index.html` | Button `#btnIaArk` `data-ia-dig-view="ark"`; host `#iaArkHost` / `#iaArkViewport` inside `#iaDigHost` |
| `js/console_ia_shell.js` | `DIG_VIEW_CLASS.ark = 'ia-dig-view-ark'`; activate/deactivate `WTM_ArkIaPanel` in `setDigView` / dig `setLayer` |
| `css/console_ia.css` | Dig visibility for ark; peer-hide BW/Midwest/A; collapsed `body.ia-left-collapsed #btnIaArk::before { content: "ARK"; }` |
| `js/ark_ia_panel.js` | UI only: table of data sources, last refreshed, status chips; listens `WTM_Ark.subscribe` |

**Dashboard content (minimum):**

- Rows: Hydration, Curve history, BBDM report (as registered), last refresh wall-clock
- Status chips: ok / warn / risk (reuse `.console-chip--*`)
- Optional “Refresh via DeskOps” button calling `WTM_DeskOps.refreshAllDeskData` (does not re-implement collect)

#### 1.3 “Fix with LLM” button

On ARK page actions bar:

1. Build diagnostic prompt via `WTM_Ark.buildFixWithLlmPrompt()` including: source inventory, last errors, stamp fields, `file:` protocol note, brief “expected paths”
2. `navigator.clipboard.writeText` + `showToast` (same pattern as `auto_collect_panel.copyText`)

### Phase 2 – Data integration

#### 2.1 `desk_data_ops.js` hosts The Ark

- Ensure Ark is loaded before DeskOps (script order)
- `refreshHydration()` becomes: **one** `WTM_Ark.loadHydration({ force: true })`, then call existing apply paths:
  1. Prefer `WTM_reloadDeployHydration` **rewired to consume Ark cache** (no second network hop), **or** pass Ark payload into `hydrateFromBundle` if cleaner
  2. BW `mergeHydrationBundle` / `reloadHydration` uses Ark cache
  3. WMC hydrate uses Ark cache
- Stamp for toasts continues from Ark `getStamp()`
- Emit existing `whinfell-desk-refresh` after apply (do not invent a second fan-out that double-paints)

#### 2.2 Rewire BasisWatch

In `js/basis_watch_panel.js`:

- Remove direct `fetch` in `loadHydrationBundle` / `ensureCurveHistory`
- Call `WTM_Ark.loadHydration` / `loadCurveHistory` (or get cache if warm)
- Keep pure analytics in `basis_watch_analytics.js` unchanged
- Standalone path still works if Ark is on the page

#### 2.3 Rewire Bang Bang DA

In `bang_bang_da_machine.html`:

- Include `js/ark.js`
- Replace `fetchFirst` / `loadReport` / hydration-meta loads with Ark APIs
- Keep presentation modules (`bbdm_ia_shell.js`, `bbdm_litmus_table.js`) free of raw I/O

**Also rewire (required by Ark absolute rule, within Phase 2):**

- `js/core.js` `reloadDeployHydration` → Ark fetch
- `midwest_compute/wmc-hydrate.js` → Ark hydration cache (merge logic stays)
- Do **not** block Phase 2 on ladder deep-dive / data_dictionary meta unless they still `fetch` hydration; if they fetch `latest.json`, route those too. Dictionary meta can remain panel-local **only if** product list doesn’t require it — **Agents.md is absolute**: any raw data file load must go through Ark. Prefer registering dictionary meta as an Ark source if it is still loaded via fetch.

### Phase 3 – Articulate (core)

#### 3.1 `js/articulate.js` engine

```text
WTM_Articulate.buildPrompt({ panelId, subject, snapshot, dataBlock })
WTM_Articulate.copyPrompt(promptText) → Promise<boolean>
WTM_Articulate.logSession({ panelId, subject, snapshot, promptPreview? })
WTM_Articulate.listSessions()
WTM_Articulate.clearSessions()   // test helper / future
```

**Prompt template** (exact structure from product plan):

```text
**WTC Analysis — [Panel / Subject]**
**Snapshot:** [Current snapshot date/time]

**Current Data:**
[Key quantitative data …]

**Instructions:**
You are an experienced floor trader turned analyst. …
```

Panel adapters (small pure functions):

- `articulate_context_basis_watch(state)` → key levels, spreads, percentiles from BW model
- `articulate_context_bbdm(report)` → top scores / litmus summary fields

#### 3.2 “A” button on major panel outputs

| Panel | Attachment |
|-------|------------|
| BasisWatch | `.bw-utility-cluster` next to CSV/PNG (`#btnBwArticulate`) |
| BBDM | Scan toolbar next to `#btnExport` |

On click: build context → `buildPrompt` → clipboard → `logSession` → toast.

#### 3.3 Session log fields (v1)

```json
{
  "id": "uuid-or-ts",
  "at": "ISO-8601",
  "panelId": "basis_watch|bbdm|…",
  "moduleName": "BasisWatch",
  "snapshot": "as_of or snapshot_id",
  "promptPreview": "first ~200 chars"
}
```

Cap list (e.g. 100 entries). Full prompt optional in log to avoid bloating localStorage — store preview + rebuildable metadata; if product requires full text for A page replay, store full prompt with a hard size cap.

#### 3.4 Full **A** page

Same dig pattern as ARK:

- `#btnIaA` `data-ia-dig-view="a_page"`, collapsed glyph **`A`**
- `js/a_ia_panel.js` → table of past sessions (timestamp, module, snapshot)
- Data only via `WTM_Articulate.listSessions()` (no raw files)

## Risks & mitigations

| Risk | Mitigation |
|------|------------|
| Double-paint on refresh | One Ark load; apply layers only; keep single `whinfell-desk-refresh` |
| Standalone BW without core | Ark script on standalone pages; apply via BW merge only |
| `file:` protocol | Ark returns clear error status; Fix with LLM prompt documents it |
| localStorage quota | Cap log entries; store preview not megabyte prompts |
| Shortcut collision | Dig tools use Alt-only; Agent remains Alt+Shift+A — use Alt+K for ARK if needed |
| docs/ drift | Prefer root + `build.sh`; update docs only if still served |
| Over-rewire scope creep | Phase 1 lands engine + UI first; Phase 2 rewires listed loaders only |
